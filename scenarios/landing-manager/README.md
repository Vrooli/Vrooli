### Example Use Case: Vrooli Pro

The first landing page generated by Landing Manager will be **Vrooli Pro** (a bundle of internal tools: browser-automation-studio, etc.). The landing page will:

1. Present different variants to visitors (A/B testing)
2. Track metrics (page views, CTA clicks, conversions)
3. Process payments via Stripe
4. Expose subscription verification endpoints that Vrooli Pro apps can query
5. Allow admins to customize content and design via the admin portal

Every future bundle (Vrooli Home, etc.) will get its own landing page scenario via the same process.

### Factory vs Template Boundaries
- **Factory UI (this scenario)**: root route `/` shows a factory dashboard with template links and generation instructions. It does **not** host production landing/admin runtime or previews.
- **Template**: `scripts/scenarios/templates/landing-page-react-vite` contains the landing/admin UI + API + requirements + PRD copied into generated scenarios.
- **Generated scenarios**: run the landing/admin experience independently on their own ports. Validate runtime features there.

**Key insight**: The registry pattern means adding a section only requires:
1. Create component: `generated/{instance}/ui/src/components/sections/{Name}Section.tsx`
2. Register in: `generated/{instance}/ui/src/components/sections/registry.tsx`
3. Update DB: Add to CHECK constraint in `initialization/postgres/schema.sql`

No switch statements, no manual type updates - it's all auto-derived from the registry!
